# Бізнес-документ проєкту "CRM для СТО"

## 1. Назва та опис системи
**Назва:** CRM-система для станції технічного обслуговування автомобілів.

Це веб-орієнтована інформаційна система, призначена для автоматизації бізнес-процесів малого автосервісу. [cite_start]Вона дозволяє централізовано зберігати інформацію про клієнтів та їхні автомобілі, вести облік наданих послуг, керувати замовленнями та контролювати фінансові операції [cite: 65-68, 80-81].

[cite_start]Впровадження системи спрямоване на вирішення проблем втрати даних про історію обслуговування, дублювання замовлень, а також на покращення взаємодії з клієнтами та зменшення операційних витрат підприємства [cite: 90-94].

## 2. Користувачі (ролі)
[cite_start]Система передбачає наступні ролі користувачів[cite: 116, 136]:
* **Адміністратор:** має повний доступ до налаштувань системи, управління користувачами (персоналом) та довідниками послуг.
* **Менеджер:** здійснює реєстрацію клієнтів та авто, оформлює замовлення, формує рахунки та комунікує з клієнтами.
* **Майстер (Механік):** переглядає призначені завдання, змінює статус виконання робіт, вносить дані про використані матеріали.

## 3. Основні сценарії використання
1.  [cite_start]**Реєстрація клієнта та авто:** Менеджер створює профіль нового клієнта і додає до нього один або кілька транспортних засобів[cite: 112].
2.  [cite_start]**Оформлення сервісного запису:** Клієнт звертається з проблемою, менеджер створює запис про обслуговування, обирає послуги з довідника та призначає виконавця[cite: 113, 218].
3.  [cite_start]**Виставлення рахунку:** Після завершення робіт система генерує рахунок (Invoice) на основі виконаних послуг, який передається клієнту для оплати[cite: 219].
4.  [cite_start]**Перегляд історії обслуговування:** Майстер або менеджер переглядає історію попередніх ремонтів конкретного автомобіля для точнішої діагностики[cite: 155].

## 4. Функціональні вимоги
* [cite_start]**Управління клієнтами:** CRUD-операції (створення, читання, оновлення, видалення) для записів клієнтів та автомобілів[cite: 112].
* [cite_start]**Управління послугами:** Ведення довідника послуг із цінами та описом[cite: 114].
* [cite_start]**Облік робіт:** Фіксація сервісних записів (дата, пробіг, виконавець, перелік робіт)[cite: 155].
* [cite_start]**Фінанси:** Формування та зберігання рахунків (Invoices) та позицій рахунків (Invoice Items) [cite: 156-157].
* [cite_start]**Автентифікація:** Реєстрація та вхід користувачів у систему[cite: 192].

## 5. Нефункціональні вимоги
* [cite_start]**Швидкодія:** Використання асинхронного фреймворку FastAPI для швидкої обробки запитів [cite: 173-175].
* [cite_start]**Масштабованість:** Архітектура на базі Docker контейнерів для легкого розгортання та масштабування[cite: 213].
* **Безпека:** Хешування паролів користувачів, розмежування прав доступу.
* [cite_start]**Зручність:** Адаптивний веб-інтерфейс (React.js), зрозумілий для персоналу без глибоких технічних знань[cite: 197].

## 6. Обмеження
* Платформа: Система є веб-орієнтованою (Browser-based). Розробка кросплатформних або нативних мобільних застосунків не входить у поточні рамки проєкту.

* Функціональні межі: Інтеграція зі сторонніми API (наприклад, складськими програмами або постачальниками запчастин) винесена за межі поточної реалізації.

* Мережева залежність: Система не підтримує роботу в офлайн-режимі (Offline-first strategy не застосовується).му: Архітектура клієнт-сервер вимагає постійного з'єднання з мережею; кешування даних для повноцінної роботи без інтернету (PWA capabilities) на даному етапі не передбачено.
